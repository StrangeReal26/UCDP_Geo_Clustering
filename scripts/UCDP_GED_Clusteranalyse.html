Info zum Herunterladen des Datensatzes

Der Datensatz wurde über das UCDP Dataset Download Center heruntergeladen (Link im nächsten Code Chunk hinterlegt).

R Studio konnte den Datensatz erst einlesen, nachdem dieser als xlsx-Datei heruntergeladen und in eine csv-Datei umgewandelt wurde. Hierbei wurde das Semikolon als Trennzeichen festgelegt.

Bei der Analyse wurde ein verkürzter Datensatz verwendet, da Spalten entfernt wurden, um die Größ der Datei zu verringern. Die Analyse funtkioniert jedoch auch mit der unveränderten Datei problemlos.

Inhalt der genutzten Variablen

Informationen stammen aus dem Codebook des GED 25.1!

“id” Eine eindeutige, dauerhafte numerische ID eines einzelnen Ereignisses. Dieselbe id bezeichnet über mehrere GED-Versionen hinweg dasselbe reale Ereignis.

“year” Das Jahr, in dem das Ereignis stattgefunden hat.

“type_of_violence” Der Typ der organisierten Gewalt nach dem UCDP: 1 = staatlicher Konflikt, 2 = nichtstaatlicher Konflikt, 3 = einseitige Gewalt gegen Zivilisten.

“conflict_new_id” Eine eindeutige Konflikt-ID für jeden einzelnen Konflikt im Datensatz. Ist geeignet zum Filtern von Konflikten.

“conflict_name” Der Name des UCDP-Konflikts, zu dem das Ereignis gehört.

“event_clarity” Eine 1 (hoch) steht für Ereignisse, bei denen der Berichterstatter das Ereignis vollständig identifizieren kann. Das bedeutet, dass das einzelne Geschehen von der Originalquelle ausreichend detailliert beschrieben wird, um einzelne Vorfälle zu identifizieren.

“latitude” Breitengrad des Ereignisortes in Dezimalgrad.

“longitude” Längengrad des Ereignisortes in Dezimalgrad.

“where_prec” Die Genauigkeit, mit der die dem Ereignis zugewiesenen Koordinaten und der Standort den tatsächlichen Standort des Ereignisses widerspiegeln. Eine 1 bedeutet, dass der genaue Standort des Ereignisses bekannt ist.

“country”
Das Land, in dem das Ereignis tatsächlich stattgefunden hat.

“region” Die Weltregion bzw. der Kontinent des Ereignisortes.

“date_start” Das früheste mögliche Datum, an dem das Ereignis stattgefunden haben kann.

“date_end” Das späteste mögliche Datum, an dem das Ereignis stattgefunden haben kann.

“deaths_civilians” Die beste Schätzung der getöteten Zivilpersonen im Ereignis.

“best”
Die beste bzw. wahrscheinlichste Schätzung der gesamten Todesopfer des Ereignisses. Sie ist die Summe aus deaths_a, deaths_b, deaths_civilians und deaths_unknown.

“high” Die höchste verlässliche Schätzung der gesamten Todesopfer des Ereignisses.

Erklärung

Aufklappen 
# HDBSCAN clustering (explorativ)


# 1) Parameter setzen: Mindestgröße/Robustheit eines Clusters

# HDBSCAN ist ein dichtebasiertes Clusterverfahren, das Cluster als Regionen hoher Punktdichte identifiziert
# Es ist eine Erweiterung des DBSCAN-Algorithmus und kann Cluster unterschiedlicher Dichte erkennen

# Im Gegensatz zu DBSCAN wird bei HDBSCAN keine Definition des Radius der Cluster (Epsilon/Eps) verwendet

# Es müssen jedoch die minPts (minimum points) definiert werden --> Die minimale Anzahl von Punkten, die in einer Nachbarschaft vorhanden sein müssen, damit eine Region als Cluster erkannt wird


# 2) HDBSCAN ausführen
# coords_m ist eine Matrix: Spalte 1 = x (m), Spalte 2 = y (m)


# 3) Ergebnisse an Datensatz anhängen

# cluster: 0 = noise (kein Cluster), 1..k = Cluster-IDs
# membership_prob: "Sicherheit" der Zuordnung (0..1)

# membership_prob beschreibt die Sicherheit, mit der ein Punkt einem Cluster zugeordnet wird
# Werte nahe 1 bedeuten eine stabile Clusterzugehörigkeit

# Die Anzahl der Cluster wird automatisch vom Algorithmus bestimmt
# Anders als bei k-means muss die Clusterzahl nicht vorgegeben werden
